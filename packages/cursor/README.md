# @bastani/cursor

First-party experimental Atomic provider for Cursor subscription models.

## Discovery, routing, and cache

Atomic uses Cursor's browser PKCE OAuth flow and private CLI-compatible HTTP/2 protocol. `POST /agent.v1.AgentService/GetUsableModels` is the sole authority for the authenticated account's runnable model existence, exact executable IDs, display data, and Max state. Atomic registers one model row for each exact returned `model_id` and sends the selected ID unchanged in both `ModelDetails.model_id` and `RequestedModel.model_id`.

`POST /aiserver.v1.AiService/AvailableModels` is a separate, best-effort metadata call for image capability only. Atomic joins its metadata to a GetUsable route only through unambiguous exact identity/variant evidence from the same authenticated account. AvailableModels cannot add or remove executable rows, choose a route, supply Max state or request parameters, or block text usage. Missing, false, or ambiguous image metadata leaves the route text-only; model-family names are not used to infer image support.

Max comes exclusively from GetUsable and is encoded in both request model structures. `RequestedModel.parameters` is always empty. Atomic does not expand AvailableModels tuples, synthesize picker or backend routes, translate a reasoning selector into another route, or resolve legacy aliases. An old or unavailable Cursor ID in settings, a CLI command, a workflow, or a restored session fails clearly and must be reselected from the current authenticated catalog; Atomic does not substitute a nearest effort, static model, AvailableModels row, another Cursor model, or another provider.

Authenticated catalogs use schema-v3, account-scoped files named `~/.atomic/agent/cursor-model-catalog.json.account-<digest>` with a 30-minute TTL. They contain only exact GetUsable-derived routes and optional same-account image flags. The digest is derived from the stable JWT subject claim, not from an OAuth token, and neither tokens nor account claims are persisted. Schema-v1, schema-v2, unscoped, and parameterized caches are ignored rather than migrated. A rotated token for the same account may reuse a fresh snapshot; another account cannot load or overwrite it. A fresh same-account v3 snapshot may keep its GetUsable-derived routes available during a temporary GetUsable failure, but stale snapshots, AvailableModels data, and static rows never become executable fallback.

Credential changes refresh immediately, while credential removal or lookup failure revokes the active authority, unregisters Cursor rows, and invalidates its scoped cache before awaited discovery fails with a sanitized login/reselection diagnostic. Superseded or out-of-order requests cannot overwrite the latest account snapshot, and future-dated timestamps are not treated as fresh. Provider-bound execution uses a live revocable authority lease to recheck the supplied credential's opaque account scope, catalog generation, TTL, and exact route synchronously at each irreversible transport or tool-result boundary, using current Max/image routing rather than retained caller metadata. When the supplied scope is also the host credential manager's currently selected account, execution atomically removes the prior account's rows and joins that account's cache activation/GetUsable discovery; stale credentials cannot switch the provider back. Each lease schedules its own TTL expiry, which actively aborts silent active and paused streams, unregisters expired rows, clears only that scoped cache and freshness state, and requires rediscovery. Publishing or revoking a catalog, changing accounts, expiry, or disposing the provider invalidates the prior lease; inbound messages are rechecked after the read race before output, multi-result resumes recheck before every write, and aborted turns are removed. A removed, fabricated, expired, disposed, or non-current-account route is rejected before transport. A first-time `/login` succeeds only after authenticated discovery registers a usable catalog for that credential scope. Discovery participates in shutdown cancellation, its failures are redacted, `atomic --list-models` waits for a required refresh, and a successful live registration does not depend on best-effort cache persistence.

Low-level consumers constructing `CursorStreamAdapter` directly must provide `executionAuthorizer` or call `bindExecutionAuthority()` with their own current-catalog validator. Each authorization carries a live assertion and abort signal for one opaque published-catalog lease; the binding is one-shot and cannot be replaced. Custom code calling `CursorConversationStateStore` turn methods directly must propagate the authorized route and opaque turn handle through register, pause, resume, cancellation, and completion so stale cleanup cannot affect a replacement turn. Message iterators receive idempotent finalization and turns detach synchronously before potentially stalled transport cleanup. HTTP/2 Run streams release local codec and open-stream bookkeeping exactly once before awaiting remote close/cancel; adapter/session disposal starts cleanup, waits only for a bounded grace period, and detaches never-settling cleanup bookkeeping so a stalled transport cannot block provider shutdown. An unbound adapter rejects execution instead of trusting caller-supplied model compatibility metadata.

## Limitations

- Cursor is experimental. Its private API may change without notice, and use may conflict with Cursor terms or account policies.
- The rewrite intentionally breaks older experimental Cursor model IDs and cache formats. Open `/model` and reselect an exact route returned for the authenticated account.
- HTTP/2 requires bundled `@bastani/atomic-natives` for the current platform.
- Credentials are OAuth-only. Do not pass Cursor tokens through arguments, environment variables, logs, or proxy processes.
- Only images on the current user turn and live mixed text/image MCP results are sent when unambiguous current-account metadata advertises image input. Earlier-turn image blocks are omitted while their surrounding text/tool continuity remains. Image data must be non-empty standard base64; MIME whitespace is accepted and removed. Unsupported or invalid current image input is rejected locally.
- Reconstructing an image sent in an earlier turn and structured clipboard attachments remain follow-up [#1807](https://github.com/bastani-inc/atomic/issues/1807). Assistant-generated images are out of scope.
