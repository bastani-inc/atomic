# @bastani/cursor

First-party experimental Atomic provider for Cursor subscription models.

## Authentication

Cursor appears in `/login` as **Cursor (Experimental)** and uses Cursor's browser PKCE flow. Atomic stores the resulting OAuth credential in the host `AuthStorage` (`~/.atomic/agent/auth.json` by default), refreshes it through the same store, and uses that exact stored OAuth access token for discovery and requests. Cursor API-key environment variables, command-line/runtime keys, `models.json` keys, provider-local token files, proxies, and credential migrations are not authentication paths for this provider. Logout, credential replacement, refresh, account changes, provider replacement, and disposal invalidate in-flight catalog/request generations.

Diagnostics are structured and redact bearer tokens and raw credential values. The catalog cache and persisted model selection contain neither credentials nor raw account identity.

## Authoritative catalog and exact selection

Authenticated `GetUsableModels` is the sole source of executable Cursor routes. A fresh, identity-matched cache of an earlier successful response may stand in for the network response; there is no static, estimated, default, synthetic, `AvailableModels`, family-derived, or previous-session fallback catalog. A valid empty response is authoritative, and one malformed row rejects the complete response rather than publishing a partial list.

Atomic preserves every response row in literal order. Route IDs are case-sensitive and unchanged, `max_mode` remains the three-state value absent/false/true, and duplicates are preserved. An occurrence is identified by the exact route ID, exact Max state, one-based occurrence within that pair, and account scope. Several occurrences may therefore share the same public `cursor/<route-id>` text. The picker adds presentation-only Max/occurrence labels; it never rewrites the public ID. Provider-plus-ID selection fails when it is ambiguous.

Settings and sessions persist a versioned provider-owned selection record containing the account scope, exact route ID, Max state, and occurrence. Restore succeeds only when those named identity fields match an exact occurrence in the current authenticated catalog. Same-version extra metadata is accepted and ignored; it never influences identity. Provider/ID-only records, unsupported versions, missing or wrong named identity fields, and unavailable runtime-only identities require explicit reselection. Atomic does not migrate, alias, normalize, or fall back to another Cursor route or provider.

## Preparation and cache

Atomic tracks required provider preparation across startup, model listing and selection, direct SDK sessions, and workflow-stage sessions. An authenticated Cursor catalog must finish current preparation before it is published. Stale completions are fenced by provider-instance, host-credential, account, client-version, catalog, request, and disposal generations and cannot publish, save, or continue delivery. An unconfigured Cursor provider remains authoritatively empty so unrelated providers can still start; losing credentials after an active catalog clears that catalog and produces an explicit authentication failure.

The token-free cache is schema version 2 at `~/.atomic/agent/cursor-model-catalog-v2.json` (or the configured Atomic agent directory). It records only a one-way account-scope digest, exact client version, fetch time, fixed 15-minute TTL, and literal route rows. Reuse requires exact schema/account/client matches and `age < 15 minutes`; equality at the TTL is expired. Empty catalogs are cacheable. Unsupported, malformed, mismatched, secret-bearing, or old records are ignored and never migrated. Writes use a mode-`0600` temporary file and atomic rename. A cache-write failure leaves the prior complete file intact and does not roll back a newly published in-memory catalog. If access and refresh values both expose accepted identity claims, those claims must match. When a stable account scope cannot be derived from otherwise valid opaque OAuth credentials, live discovery still works for an explicitly selected current-session exact route, but disk caching and selection persistence are disabled; those routes are never an implicit first/default and a restore that requires persistence reports the limitation.

## Text transport

Before transport, Atomic requires the adapter to hold a current catalog-backed route-authority lease and validates that the selected model still carries that exact route reference. Adapters without authority and fabricated or stale references fail before `Run`. `AgentRunRequest.model_details` and `AgentRunRequest.requested_model` receive the same exact route ID. `ModelDetails.max_mode` preserves absent/false/true; `RequestedModel.max_mode` maps absent to `false` only because that checked-in proto3 bool cannot represent presence. `RequestedModel.parameters` is semantically `[]` and has zero field-3 elements on the wire.

The provider streams text and thinking deltas and emits one Atomic terminal event only when one clean Connect end-stream is followed by EOF. Provider `done` messages are withheld until that terminal validation; after a published tool call, later boundary messages are held until its results resume the stream, so an unterminated EOF cannot be hidden by an early Atomic success or paused-turn handoff. A separate 100 ms tool-batch idle policy emits the bounded `toolUse` handoff when a live provider remains open awaiting its result; setting the general request/read `timeoutMs` to `0` does not disable this handoff. Malformed EOF or another transport error detected during that validation interval wins, produces one structured failure, and retains no paused turn. Before successful completion is observable, Atomic stops the heartbeat interval, waits for every already-started heartbeat write, gracefully closes native input or abandons the raw handle when close rejects, disposes codec run state, and releases the lifecycle slot exactly once. Bare EOF without a valid end-stream—including an empty frame iterable—is a structured malformed-protocol failure; heartbeat, raw-handle, codec, and lifecycle ownership are released exactly once before rejection is observable. Repeated or concurrent cancellation/pump-failure cleanup shares one raw-handle terminal operation, malformed runs choose one codec discard/dispose action, and success is never reclassified as cancellation. Run HTTP authentication statuses reject before stream acceptance, and any frame after a clean end-stream rejects as malformed. Every stream-pump failure clears the heartbeat and closes/cancels native ownership before exposing its structured error. Cancellation or route-generation invalidation does the same, including for a signal-ignoring stream returned after its route became stale but before turn registration, and never switches route, mode, or provider. Request authentication revalidates the exact credential after host synchronization or refresh; logout remains a structured missing-authentication failure. Structured redacted errors distinguish authentication, discovery, malformed protocol/frame, transport, timeout, server, cancellation, and stale-generation failures.

Each request rebuilds canonical user text, assistant text/thinking, tool calls, and text tool results from Atomic's message context, including continuation, persisted restart, and workflow-stage sessions. Historical and live-result text blocks, including empty user/assistant/thinking blocks, remain verbatim and ordered without inserted separators when image blocks are omitted. Duplicate live tool-call IDs preserve FIFO occurrence/result correlation. An orphan text tool result is retained as text without inventing a tool call. Process-local checkpoint/blob state is an optimization and is not allowed to replace, truncate, or reorder canonical history.

Atomic advertises ordinary host tools through the minimal request-context/MCP bridge, executes those tools itself, and sends text results back. Cursor-native shell/read/write/delete, MCP-resource, screen, computer-use, diagnostics, and other native execution requests receive typed safe rejections rather than execution; diagnostics uses a correlated MCP error because its matching result message is empty. Transport is direct HTTP/2 through the bundled `@bastani/atomic-natives` Rust/N-API client; there is no localhost or compatibility proxy.

## Text-only and accounting bounds

Every Cursor route advertises `input: ["text"]`. A current user image fails before the Run; a live image tool result fails before that result content is transported. Historical image blocks are omitted while adjacent historical text remains in order. Image support is follow-up scope; model-family names do not enable it.

All routes use the same conservative Atomic accounting bounds: a 200,000-token context window and 64,000-token maximum output. These are host budgeting constants, not limits advertised by Cursor, and they do not affect catalog membership, identity, capabilities, selection, persistence, or wire routing.

## Private API and verification status

Cursor uses private, undocumented endpoints and Cursor CLI-compatible headers. The protocol may change without notice, and use may conflict with Cursor's terms or account policies. The checked-in protocol descriptors and deterministic sanitized fixtures are evidence for the implemented fields, not proof that the live service has not drifted.

One bounded live Cursor preparation attempt was made during verification and timed out before `Run`; it produced no successful live protocol capture. No credential refresh completed. The fixture corpus is synthetic and contains no credential, cookie, account identifier, request ID, prompt, machine path, customer content, or retained private response. See [`src/proto/README.md`](src/proto/README.md) for the field-level evidence ledger and verification status.
